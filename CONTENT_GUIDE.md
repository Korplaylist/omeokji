# 오먹지 콘텐츠·이미지 가이드

## 핵심 원칙

레시피 이미지는 장식이 아니라 조리 설명의 일부다. 본문에 없는 재료를 이미지에 추가하거나, 앞 단계에 없던 재료가 설명 없이 다음 단계에 등장하면 안 된다.

## 연속 이미지 체크리스트

1. 첫 이미지에서 조리에 사용할 핵심 재료를 모두 보여준다.
2. 다음 단계에서도 같은 재료의 색, 크기, 손질 상태가 자연스럽게 이어져야 한다.
3. 조리도구, 그릇, 식탁, 카메라 각도, 빛의 방향과 색감을 한 글 안에서 고정한다.
4. 재료가 추가되거나 제거되는 순간은 본문과 이미지 양쪽에서 설명한다.
5. 완성 사진은 이전 단계에서 사용한 재료만으로 구성한다.
6. 별도 활용 레시피는 같은 원재료와 비주얼 시스템을 유지하되, 각 레시피 본문에 언급된 부재료만 사용한다.
7. 이미지마다 구체적인 `alt`와 단계 연결을 설명하는 캡션을 작성한다.

## 상세 레시피 작성 기준

1. 만드는 순서의 모든 번호에는 그 번호의 핵심 행동과 정확히 일치하는 이미지를 한 장씩 배치한다.
2. 만드는 순서는 4단계 이상으로 나누고, 한 단계에는 한 가지 핵심 행동만 설명한다.
3. `먹기 좋은 크기`, `적당히 익힌다`처럼 모호하게 쓰지 않는다. cm 크기, 분·초, 불 세기와 완료 상태를 함께 적는다.
4. 뼈 제거, 칼질, 접기처럼 처음 하는 사람이 어려워할 행동은 손의 위치와 확인 방법까지 설명한다.
5. 과정 이미지의 재료 개수와 손질 상태는 본문 수치와 일치시킨다.
6. 각 메뉴에 실패 방지 팁을 하나 이상 넣고, 실패 원인과 수정 방법을 함께 적는다.
7. 구조화 데이터의 조리 단계도 화면에 보이는 최신 순서와 일치시킨다.
8. 활용백서 한 글에 여러 요리가 포함되면 각 요리를 `publishedMenus`에 별도 메뉴로 등록한다. 이 공통 목록에서 홈의 `많이 찾는 한 끼`와 `오늘 뭐 먹지` 추천을 함께 생성한다.
9. 단일 레시피 본문에는 `대체 재료`, `보관과 재가열`, `실패 해결`, `익힘·완성 기준`을 요리에 맞게 구체적으로 작성한다.
10. 위 실전 가이드는 재료명·중량·시간·상태 기준을 해당 레시피에 맞춰 작성하고 다른 글의 문장을 복사하지 않는다.

## FAQ 작성 기준

1. 모든 글 마지막 부분에는 해당 요리에만 적용되는 FAQ 3개를 작성한다.
2. 질문은 대체 재료, 실패 원인, 익힘 확인, 식품 안전, 보관·재가열 중 검색자가 실제로 궁금해할 주제로 구성한다.
3. `재료의 양을 늘려도 되나요?`처럼 어느 요리에나 붙일 수 있는 공통 질문을 반복하지 않는다.
4. 답변에는 해당 요리의 재료명, 시간, 온도, 보관 기한 또는 눈으로 확인할 수 있는 상태를 최소 하나 포함한다.
5. 새 글 발행 전 기존 글 FAQ와 문장 중복 여부를 확인한다.

## 발행일 관리 기준

1. `datePublished`, `dateModified`와 화면의 작성일은 운영자가 요청한 날짜만 사용한다.
2. 콘텐츠 추가나 디자인 수정만으로 발행일을 임의 변경하지 않는다.
3. 날짜 변경 요청이 없으면 기존 발행일과 수정일을 그대로 유지한다.

## 제휴 박스 필수 기준

1. 모든 레시피와 활용백서에는 Q&A 바로 뒤에 `관련 재료 보기` 박스를 한 개 배치한다.
2. 단일 레시피는 핵심 재료 2~4개, 활용백서는 포함된 메뉴 전체를 포괄하는 재료 검색어를 사용한다.
3. 외부 상품 링크에는 `rel="sponsored noopener"`를 사용하고 새 창으로 연다.
4. 박스 하단에는 `쿠팡 파트너스 활동을 통해 일정액의 수수료를 제공받을 수 있습니다.` 문구를 항상 표시한다.
5. 쿠팡 API 연동 전에는 검색 링크를 사용하고, API 연동 후에도 재고·가격을 보장하는 표현은 사용하지 않는다.

1. 제목 맨 앞에는 사용자가 실제로 검색할 핵심 요리명과 검색 의도를 둔다. 예: `두부김치 황금레시피`, `남은 치킨 활용 레시피`.
2. 뒤에는 양념 비율, 핵심 재료 용량, 조리 시간, 조리도구처럼 본문에서 실제로 제공하는 구체적 효익을 하나만 붙인다.
3. 감성 문구만으로 제목을 만들거나 `대박`, `무조건`, `역대급`처럼 검증할 수 없는 과장 표현을 사용하지 않는다.
4. `<title>`, H1, `og:title`은 표현을 조금 줄일 수 있지만 핵심 검색어와 효익은 동일하게 유지한다.
5. 브랜드명 `오먹지`는 검색 제목의 마지막에만 붙인다.

## 홈 성능·접근성 발행 기준

1. 홈 카드에 등록하는 모든 이미지는 `-320.webp`와 `-640.webp` 두 크기를 함께 생성한다.
2. 카드 `<img>`에는 `srcset`, `sizes`, `width`, `height`, `loading="lazy"`, `decoding="async"`를 빠짐없이 넣는다.
3. 320px 이미지는 20KiB 이하를 목표로 압축하고 원본 이미지를 홈 카드에 직접 사용하지 않는다.
4. 레시피 목록은 `<ul>`과 `<li>`로 구성한다. `article` 요소에 `role="listitem"`을 강제로 지정하지 않는다.
5. 홈의 전체 CSS는 렌더링 차단 없이 불러오고, 첫 화면에 필요한 최소 스타일만 문서 안에 유지한다.
6. 새 메뉴와 이미지를 추가한 뒤 반드시 `npm run qa`를 통과시킨 뒤 배포한다.
7. GitHub 배포 작업에서도 동일한 품질 검사를 실행하며 실패한 상태로 Cloudflare 배포를 진행하지 않는다.

### 홈 성능 재발 방지 규칙

1. 레시피 카드는 PC용 `-220.webp`와 모바일용 `-320.webp`를 생성한다. `<picture>`와 `(min-width:681px)` source로 PC에는 화면 밀도와 무관하게 220px만 전송하고 모바일에는 320px를 전송한다. 640px 후보는 추가하지 않는다.
2. 캐러셀 JavaScript에서 `offsetWidth`, `clientWidth`, `scrollWidth`, `getBoundingClientRect()`를 사용하지 않는다. 카드 인덱스와 `scrollIntoView()`로 이동한다.
3. 모든 레시피 카드 이미지는 음식명을 포함한 비어 있지 않은 `alt`를 작성한다.
4. 위 규칙은 `npm run qa`에서 검사하며 실패하면 Cloudflare 배포를 중단한다.
5. 홈은 전체 CSS를 인라인하여 외부 스타일의 후적용에 따른 강제 리플로우를 방지한다. `styles.css`를 수정한 뒤 반드시 `npm run sync:home-css`를 실행하며, 동기화되지 않으면 QA와 배포가 실패한다.

### 신규 레시피 발행 체크리스트

1. 독립 레시피는 정확한 1인분 재료량, 최소 4개의 상세 조리 단계, 각 단계와 일치하는 연속 이미지, 실패 방지, 대체 재료·보관·익힘 기준을 포함한다.
2. 글 마지막에는 메뉴별 Q&A 3개, 관련 재료 보기, 제휴 고지, 추천 레시피 5개를 포함한다.
3. 모든 이미지에는 설명형 `alt`를 작성하고 화면에는 `ⓒ오먹지`만 오른쪽 정렬로 표시한다.
4. Recipe, FAQPage, BreadcrumbList 구조화 데이터와 canonical, OG 메타데이터, 사이트맵 URL을 함께 등록한다.
5. 홈 메뉴 데이터에 제목·요약·태그·URL을 등록하고 목록용 220px·320px 이미지를 함께 생성한다.
6. 야식 필터는 현재 총 7개를 유지하며 `npm run qa`가 글 수와 필수 구성요소를 검사한다.
7. 상단의 레시피와 활용백서는 홈 내부 앵커가 아닌 각각 `/recipes.html`, `/guides.html` 전용 목록으로 이동한다. 모든 상세 글도 같은 전용 페이지 링크를 사용하는 공통 상단 메뉴를 유지한다.
8. 메뉴나 활용백서 글을 추가한 뒤 `npm run build:directories`를 실행하고, 전용 페이지의 전체 항목 수가 QA를 통과해야 배포한다.
9. 홈의 작은 레시피 카드는 화면 밀도에 따라 220px 또는 320px를 선택하고, 전용 레시피 목록의 큰 카드는 320/640px 반응형 이미지를 사용한다. 활용백서 전용 목록은 과압축한 홈 표지가 아닌 원본 640px 표지를 사용한다.
